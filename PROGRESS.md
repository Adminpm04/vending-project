## Что сделано

### Backend (FastAPI) — `backend/`
- `main.py` — kiosk API (каталог, покупка), WebSocket контроллеров (auth по machine_id+токен),
  поллинг JetQR, dispense-флоу с авто-возвратом при сбое, admin API (точки, слоты, сессии,
  статистика, blacklist, ручная выдача, повтор возврата).
- `jetqr.py` — создание/проверка/отмена инвойсов. **Исправлено под боевую доку** (см. раздел JetQR).
- `database.py` — таблицы `VendingMachine`, `ProductSlot`, `VendingSession`, `Blacklist`.
- `vmc_protocol.py` — протокол RS232 XY V3.0, проверен тестами против пакетов из PDF.
- `config.py` — настройки из `.env`.

### Контроллер точки — `controller/controller.py`
- RS232 POLL/ACK-цикл (ответ VMC в пределах 100 мс, ретраи до 5 раз) + WebSocket к серверу.
- Выдача командой **`0x06`** (drive-direct, drop_sensor=1, elevator=0) — ровно как рабочее
  приложение DC (не `0x03`).

### Frontend — `frontend/`
- `kiosk.html` — экран клиента: каталог → QR → статус оплаты/выдачи.
- `admin.html` — админка: точки, слоты/цены, сессии, возвраты, выручка.

## Железо (проверено через adb + разбор APK)

Планшет: фирменный Android 12 (`Bengal for arm`, сборка `XY9186_Android12_V1.02_EM test-keys`),
**разлочен, root не нужен**. Отладка по USB и по Wi-Fi включены.

| Что | Значение |
|---|---|
| Порт VMC | **`/dev/ttyS1`** (ttyS2 свободен, ttyS0 нет) |
| Скорость | **57600 8N1** |
| Root нужен? | **Нет** — порт с правами `crwxrwxrwx`, открыт любому приложению |
| Команда выдачи | **`0x06`** drive-direct, drop_sensor=1, elevator=0 |
| XOR-чексумма | XOR всех байт `& 255` (совпадает с нашим `xor_checksum`) |

Установленные приложения на планшете:
- `com.xyshj.machine` — фирменное настроечное приложение XY (для техника, оставляем).
- `dc.com.vending` — **рабочий референс** (Gorilla/DC-кошелёк). Разобран androguard'ом.
- `com.sugarapps.autostartmanager` — автозапуск (тот же приём применим нам).

**Референс DC подтвердил нашу архитектуру 1:1:** товары/цены тянутся с центрального сервера
по серийнику (`GET .../v2/getproducts1?sn=<serial>`), выдача через `/dev/ttyS1`. Отличие
только в платёжке (у них DC-кошелёк, у нас JetQR). APK лежат: на сервере `/tmp/dc_vending.apk`,
`/tmp/xyshj_machine.apk`.

**Важно:** тестовый планшет сейчас НЕ подключён к автомату → полный тест выдачи невозможен,
пока планшет не вернут в автомат. Планшеты одинаковые — что настроим, то поедет в автомат.

## JetQR / оплата — ТЕКУЩИЙ БЛОКЕР

**Статус:** реквизиты вписаны, но эндпоинт даёт **401 Unauthorized**. Оплата не работает.

Диагностика (точная):
- Правильный эндпоинт: `https://dev-jetqr.aliftech.net/test/api/v1/merchant/invoice`
  (возвращает 401, а не 404 → путь и base URL верные, наш config корректен).
- Мерчантская авторизация — **только `X-Api-Key`**, HMAC-подпись НЕ нужна (по офиц. доке
  раздел ExternalMerchantInvoice). Подпись как причина — исключена.
- 401 сохраняется даже с идеально правильным телом → **проблема чисто в ключе/среде**.
- Реквизиты `merchant 23174 / store 31494 / terminal 36051 / api_key 0514b40d-...` — это
  ТЕ ЖЕ, что у parking-проекта (боевые ZT). Вписаны «для теста».

**Что нужно от Aliftech (это разблокирует оплату):**
1. Рабочая пара **ключ + правильный base URL**: эти ключи боевые или тестовые? По какому
   адресу ими обращаться?
2. Если 0514... боевой — дать **тестовые ключи** под dev-среду (тест без реальных денег).
3. Для production вендинга — **свои store_id/terminal_id** (по одному на автомат), чтобы
   выручка не смешивалась с парковкой.

Исправлено в `jetqr.py` по боевой доке (чтобы заработало сразу, как дадут ключ):
- Успех создания = `code==200` + `invoice_id` (поля `type` в ответе нет — старая проверка
  `type==SUCCESS` не срабатывала бы никогда).
- Добавлено обязательное поле `mis_payment_time` (ISO 8601).
- `code` в проверке статуса читается строкой (`"200"`/`"202"`/`"203"`/`"404"`).
- На будущее: JetQR поддерживает **callback** (`invoice.in_process`) — лучше поллинга,
  требует публичный HTTPS.
