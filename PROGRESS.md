## Диагностика зависаний выдачи (2026-07-15)

Со вчерашнего утра (14.07 ~08:00) все выдачи на VND-002 стабильно проваливались
(164 failed сессий подряд, последняя удачная — 14.07 06:17). Разобрались вживую на
VND-003 (второй тестовый автомат, планшет подключён по Tailscale VPN к серверу —
`tailscale status` на сервере покажет его как `msm8953-for-arm64`, adb доступен
удалённо через `adb connect <tailscale-ip>:5555`).

**Причина №1 — неправильная нумерация слотов.** Наша БД хранит `slot_id` как простые
числа `1-8`. Реальный VMC на этой модели адресует селекции трёхзначными кодами
«ряд+позиция»: `101-111, 201-211, 301-311, 401-411, 501-511` (5 рядов × 11 позиций).
Числа `1-11` тоже существуют как отдельный диапазон и отвечают на `check-slot`, но
это не настоящие лотки с товаром — выдача по ним стабильно клинит («Selection
jammed», автомат сам ставит их в паузу после сбоя). Настоящую карту слотов узнали
через заводское приложение **`com.xyshj.machine`, ярлык "XY VMC"** (`.activity.SplashActivity`,
пароль не нужен — это открытый каталог, в отличие от запароленного экрана настроек
того же пакета) — оно тянет данные прямо с VMC и показывает реальные загруженные
позиции постранично. Важно: пока это приложение открыто, оно держит `/dev/ttyS1`, и
наш APK не может к нему достучаться (и наоборот) — параллельно работать не могут,
нужно `force-stop` одного перед запуском другого.

**Причина №2 (главная, починена) — баг в определении успеха.** У этой прошивки VMC
финальный статус выдачи (`0x04`, код `0x02`/`0x24` = успех) в поле "selection"
возвращает **`0`** вместо настоящего номера слота (напр. `101`). Наша защита от
«эха» старого запроса (`VmcController.kt` / `controller.py`, ждём статус именно по
своему `slot_id`) считала это чужим/устаревшим ответом и отбрасывала — то есть
**реальная успешная выдача репортилась в бэкенд как `VMC response timeout`**.
Живой тест подтвердил: force-dispense на слот 101 реально выдал товар (статусы
`0x10→0x01→0x11→0x02→0x24`), но из-за бага результат ушёл как таймаут.

Фикс: слот-матчинг теперь принимает `slot == 0` как валидное завершение (не только
точное совпадение). Подтверждено живым тестом — `force-dispense` на свежий слот 102
после фикса корректно вернул `{"success": true, "code": 2}`.

- Коммит: `2fdc75f` (в `dev`), PR в `main`: https://github.com/Adminpm04/vending-project/pull/1
- Новый APK собран через GitHub Actions (`android-build.yml`) и вручную поставлен на
  VND-003 (adb install — потребовался uninstall/reinstall, у CI-сборки не совпала
  debug-подпись с предыдущей установкой; настройки (machine_id/token/server_url)
  сохранены и восстановлены вручную через `run-as ... shared_prefs`).
- **VND-002 всё ещё на старой (багованной) версии APK и со старыми номерами слотов
  `1-8`** — не тронуто, см. «Следующие шаги».
- Автообновление APK (`UpdateChecker.kt`) сверяет sha256 `frontend/vending-kiosk.apk`
  на сервере и **само не устанавливает** — только скачивает и открывает системный
  экран установки, тап «Установить» нужен вручную на планшете. Обновлять файл на
  сервере, пока VND-002 в проде без присмотра — риск: если/когда точка снова
  подключится, на экране киоска всплывёт установщик, который никто не подтвердит.

## Что сделано

### Backend (FastAPI) — `backend/`
- `main.py` — kiosk API (каталог, покупка), WebSocket контроллеров (auth по machine_id+токен),
  поллинг JetQR, dispense-флоу с авто-возвратом при сбое, admin API (точки, слоты, сессии,
  статистика, blacklist, ручная выдача, повтор возврата).
- `jetqr.py` — создание/проверка/отмена инвойсов. **Исправлено под боевую доку** (см. раздел JetQR).
- `database.py` — таблицы `VendingMachine`, `ProductSlot`, `VendingSession`, `Blacklist`.
- `vmc_protocol.py` — протокол RS232 XY V3.0, проверен тестами против пакетов из PDF.
- `config.py` — настройки из `.env`.

### Контроллер точки — `controller/controller.py`
- RS232 POLL/ACK-цикл (ответ VMC в пределах 100 мс, ретраи до 5 раз) + WebSocket к серверу.
- Выдача командой **`0x06`** (drive-direct, drop_sensor=1, elevator=0) — ровно как рабочее
  приложение DC (не `0x03`).

### Frontend — `frontend/` (редизайн по макету Claude Design, 2026-07-03)
Единый премиум-стиль: шрифт Manrope, цвета #111111/#8B253B/#F5F5F5, 3D-подложки,
светлая и тёмная тема (по `prefers-color-scheme` + переключатель). Продакшн
HTML/CSS/JS без сборки (FastAPI отдаёт статику; работает в web и Android WebView).
Проверено jsdom-рендер-тестом: все экраны/вкладки, обе темы — без JS-ошибок.
- `kiosk.html` — экран автомата, fullscreen: K1 каталог (3D-карточки, онлайн-пилюля,
  часы) → K2 оплата (QR + таймобр отсчёт) → K3 выдача → K4 успех → K5 возврат →
  K6 недоступен. Реальный QR из `/api/qr`, WebSocket-статусы + поллинг.
- `admin.html` — вход по токену; сайдбар (desktop) + нижняя навигация (mobile,
  через CSS media-queries); тосты. Вкладки на реальных данных: Дашборд (метрики,
  график выручки 14 дней, последние продажи), Точки (список + схема-карта,
  добавление), Слоты точки (правка цены/остатка на месте, добавление), Сессии
  (фильтры, повтор возврата), Товары (каталог из слотов), Статистика (часы пик,
  топ товаров, экспорт CSV), Настройки (тема, язык, чёрный список).

| Команда выдачи | **`0x06`** drive-direct, drop_sensor=1, elevator=0 |

Установленные приложения на планшете:
- `com.xyshj.machine` — фирменное настроечное приложение XY (для техника, оставляем).
- `dc.com.vending` — **рабочий референс** (Gorilla/DC-кошелёк). Разобран androguard'ом.
- `com.sugarapps.autostartmanager` — автозапуск (тот же приём применим нам).

**Референс DC подтвердил нашу архитектуру 1:1:** товары/цены тянутся с центрального сервера
по серийнику (`GET .../v2/getproducts1?sn=<serial>`), выдача через `/dev/ttyS1`. Отличие
только в платёжке (у них DC-кошелёк, у нас JetQR). APK лежат: на сервере `/tmp/dc_vending.apk`,
`/tmp/xyshj_machine.apk`.

**Важно:** тестовый планшет сейчас НЕ подключён к автомату → полный тест выдачи невозможен,

верный адрес: **`https://jetqr.aliftech.net`** (без `dev-`, без `/test`). Подтверждено живым
тестом — invoice создался, статус опрашивается, оба ответа 200 OK.

**Текущий конфиг (`.env` на сервере, значения не в git — сами ключи см. только там):**
- `JETQR_BASE_URL=https://jetqr.aliftech.net`
- `JETQR_MERCHANT_ID=23174`, `JETQR_STORE_ID=31494`
- `JETQR_TERMINAL_ID=36050` (`JETQR_API_KEY` — соответствующий ключ в `.env`)
  — это **касса 2** из трёх присланных Alif-ом (ZT Parking-In кассы 1/2/3, одолжены у
  Parking-проекта временно). Касса 3 (`36051`) — та, что уже была в конфиге и стабильно

  «рабочей» вслух в чате, но текстом не подтверждали — не пробовали.

**Важно, до конца не закрыто:**
1. Выручка с вендинга сейчас падает в **тот же мерчант-аккаунт, что и парковка** (смешаются
   отчёты Alif-у). Для постоянной работы всё ещё нужны **свои store_id/terminal_id** под
   вендинг — попросить у Aliftech отдельно.
2. `cancel_invoice` (возврат при сбое выдачи) с этим боевым адресом/ключом **не тестировался**

3. Ключи — временно занятые, Alif может их отозвать/переактивировать под парковку в любой

Исправлено в `jetqr.py` по боевой доке (чтобы заработало сразу, как дали ключ):
- Успех создания = `code==200` + `invoice_id` (поля `type` в ответе нет — старая проверка

- `code` в проверке статуса читается строкой (`"200"`/`"202"`/`"203"`/`"404"`).
- На будущее: JetQR поддерживает **callback** (`invoice.in_process`) — лучше поллинга,

## Авторизация админки — переделана (2026-07-05)

Общий `ADMIN_TOKEN` заменён на полноценных пользователей: логин/пароль,
серверные сессии (Bearer-токен), три роли — `admin` (всё, включая
пользователей и чёрный список), `operator` (точки/слоты/сессии/возвраты,
без настроек), `viewer` (только просмотр). Стартовый admin-аккаунт создаётся
один раз из `ADMIN_BOOTSTRAP_USERNAME`/`ADMIN_BOOTSTRAP_PASSWORD` (.env на
сервере) при пустой таблице `admin_users` — дальше пользователей заводит сам
admin через «Настройки → Пользователи панели». Пароли — PBKDF2-HMAC-SHA256
(stdlib `hashlib`, без новых зависимостей). Роли проверяются и на backend
(403 при недостатке прав), и на frontend (скрытие недоступных разделов) —
задеплоено и проверено на сервере (login/me/users CRUD/RBAC 403 — все ок).

## Каталог товаров + цены по точкам (2026-07-05)

Вкладка «Товары» — полноценный каталог: добавление/редактирование/удаление
товаров, загрузка фото (файл → превью → `/api/admin/upload`, base64→файл в
`/opt/vending-project/uploads`, отдаётся по `/uploads`), категория и цена по
умолчанию. Товар можно продавать на многих точках с РАЗНОЙ ценой на каждой:
цена хранится на слоте (per-point), редактируется и в карточке точки, и в
«Товары → Цены по точкам». В форме слота — выбор товара из каталога
(подставляет имя+фото+дефолтную цену) либо свободное название. Фото/имя
товара при изменении распространяются на все привязанные слоты, kiosk
показывает фото. Модель: таблица `Product` + `ProductSlot.product_id` (FK,
nullable; имя/фото денормализованы на слот). Проверено на сервере end-to-end.

## Аудит-проход по всей системе (2026-07-08)

Полная ревизия backend + kiosk + admin + controller + Android-мост, исправлено:
1. **WS heartbeat мог умереть навсегда** (ValueError при гонке отключения киоска) —
   автоматы молча уходили бы в офлайн через NAT-таймауты.
2. **«Отмена» на киоске теперь отменяет сессию и на сервере** — новый
   `POST /api/kiosk/session/{id}/cancel` (только pending); раньше поздний платёж
   выдал бы товар при сброшенном экране.
3. **Чёрный список**: добавлены GET/DELETE + кнопка «Разблокировать» в админке
   (раньше список был write-only и после перезагрузки казался пустым).
4. **XSS в kiosk.html** — имя/фото товара теперь экранируются.
5. Чистые 400/409 вместо 500: создание точки (пустой/дублирующийся machine_id,
   формат ограничен `[A-Za-z0-9_-]{1,50}`), blacklist, покупка без slot_id,
   мусор в ценах слотов/товаров.
6. `jetqr.cancel_invoice` — код ответа нормализуется строкой (как в check_invoice).

Проверено: живой API-тест на SQLite (25 проверок, все зелёные) + jsdom-тест обоих
фронтов (XSS, флоу покупка→отмена, все вкладки, чёрный список). Скрипты тестов —
одноразовые, в репо не входят.

Известное ограничение: график «Выручка за 14 дней» строится из последних 200
сессий — при росте продаж начнёт занижать старые дни (нужен эндпоинт агрегации).

1. Админка → «Точки» → «+ Добавить точку» → machine_id (напр. `VND-002`), название, локация.

3. На планшете: долгий тап в левом верхнем углу → вписать ID автомата и токен;
   адрес сервера (`http://10.251.4.253:8000`) и порт (`/dev/ttyS1`) — дефолты верные → «Сохранить».
4. APK качается прямо на планшете по ссылке `http://10.251.4.253:8000/apk`.
5. **До заведения слотов в админке** — открыть на планшете заводское приложение
   "XY VMC" (`com.xyshj.machine`, ярлык на главном экране, без пароля) и выписать
   реальные номера загруженных позиций с остатками. Не считать, что нумерация 1-8 —
   на этой модели автомата реальные селекции трёхзначные (`101-111, 201-211, ...`),
   см. «Диагностика зависаний выдачи» выше. Завести слоты в админке с этими
   номерами, а не по порядку 1,2,3...
