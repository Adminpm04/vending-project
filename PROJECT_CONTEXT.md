## 1. Что за проект

Система приёма оплаты через единый QR-код (национальный интероперабельный QR-стандарт,
оператор — **JetQR / Aliftech**, домен `dev-jetqr.aliftech.net`) на вендинговых автоматах,
установленных на нескольких точках по городу, с автоматической выдачей выбранного товара
и централизованной статистикой по всем точкам.

Клиент подходит к автомату → выбирает товар на экране → появляется QR (с точной суммой
этого товара) → сканирует любым банковским приложением → оплачивает → автомат выдаёт товар.
Если автомат физически не смог выдать товар — деньги возвращаются автоматически.

## 2. Референсный проект (архитектурная основа)

Есть действующий проект **`adminpm04/parking-project`** (оплата парковки через тот же
JetQR + открытие шлагбаума камерами Dahua). Архитектура вендинг-проекта **переносится оттуда
1:1**, с заменой "камера + шлагбаум" на "экран автомата + выдача товара". Ключевые файлы
оттуда для референса:

- `backend/jetqr.py` — создание и проверка инвойса JetQR (переносится почти без изменений).
- `backend/main.py` — FastAPI-роуты, WebSocket-уведомления, поллинг оплаты (`poll_payment`).
- `backend/database.py` — модели SQLAlchemy (`ParkingSession`, `Abonement`, `Blacklist`).
- `backend/dahua.py` — пример модуля управления железом (заменяется на `vending.py`).

## 3. Производитель автоматов

**Hunan Xingyuan Technology Co., Ltd.** (бренд **XY-Vending**, xy-vending.com).
Модель на фото: **XY-SLY-5C-002BL** (комбо-автомат, напитки на спиральных рядах, 220В/650Вт).

У производителя есть собственное приложение/облачная платформа ("智慧新零售云平台" — Smart
New Retail Cloud Platform) для удалённого управления автоматом (мониторинг, инвентарь), но
это инструмент **для оператора точки**, не публичный API для сторонних платежей — мы его
не используем, интегрируемся напрямую с платой автомата.

## 4. Протокол интеграции с автоматом — ключевая находка

Производитель предоставил официальный документ **"VMC - Upper computer" (V3.0)** — протокол
связи между платой автомата (VMC) и внешним контроллером ("Upper computer" — в нашем случае
это будет наш Raspberry Pi / ARM-плата на точке). Это открытый, документированный протокол,
**не MDB**, гораздо проще в реализации.

### 4.1. Физика и транспорт
- Интерфейс: **RS232**, асинхронный.
- Параметры: **57600 бод, 8 data bits, 1 stop bit, no parity**.
- VMC — хост (master), Upper computer — слейв. VMC опрашивает (POLL) каждые **200 мс**.
- Upper computer должен ответить на POLL в течение **100 мс**.
- При отсутствии ACK — до 5 повторных попыток отправки с обеих сторон.

### 4.2. Формат пакета
```
STX (2 байта) | Command (1 байт) | Length (1 байт) | PackNO+Text (n байт) | XOR (1 байт)
```
- `STX` = `0xFA 0xFB` (начало пакета).
- `Length` = длина `PackNO+Text`.
- `XOR` = XOR-контрольная сумма от STX до Text включительно.
- Пример пакета POLL: `0xFA 0xFB 0x41 0x00 0x40`.
- Пример пакета ACK: `0xFA 0xFB 0x42 0x00 0x43`.
- Номер коммуникации (PackNO) — от 1 до 255, увеличивается на 1 после каждого успешного обмена,
  одинаковый при повторных попытках отправки одного и того же пакета.

### 4.3. Минимальный набор команд для выдачи товара (главное!)

Документ прямо говорит: **достаточно 3 команд**, чтобы выдавать товар, при этом
**цену и остатки товара ведёт наше приложение**, а не VMC:

| Команда | Код | Кто шлёт | Назначение |
|---|---|---|---|
| Upper computer selects to buy | `0x03` | мы → VMC | Выбрать слот на выдачу (Text = номер слота, 2 байта) |
| VMC dispensing status | `0x04` | VMC → мы | Статус выдачи в реальном времени |
| Upper computer checks selection working | `0x01` | мы → VMC | Проверка исправности слота перед покупкой (опционально) |

**Коды статуса `0x04` (ответ VMC на выдачу) — по ним строится refund-логика:**
- `0x01` Dispensing (идёт выдача)
- `0x02` **Dispensing successfully** — успех, закрываем сессию
- `0x03` Selection jammed — застряло → возврат денег
- `0x04` Motor doesn't stop normally → возврат денег
- `0x06` Motor doesn't exist → возврат денег
- `0x07`/`0x10`-`0x13` ошибки лифта/элеватора → возврат денег
- `0xFF` Purchase terminated

(Полный список кодов ошибок — в приложенном PDF `VMCUpper_computer_V3.0_0411.pdf`,
раздел 4.3.3 — обязательно перенести файл в репозиторий вендинг-проекта, в `docs/`.)

### 4.4. Остальные команды документа (для справки, не обязательны для MVP)
- `0x11`–`0x17` — обмен по слотам: цена, остатки, ёмкость, ID товара (можно игнорировать,
  т.к. это ведём в своей БД).
- `0x21`, `0x23`–`0x28` — обмен по оплате наличными/картой на самом автомате (не используем,
  т.к. вся оплата идёт через наш JetQR-флоу).

## 5. Интеграция с JetQR

### 5.1. Базовый URL
`https://dev-jetqr.aliftech.net/api/v1` (тест), swagger: `/swagger/index.html`.

### 5.2. Аутентификация
- Заголовок `X-Api-Key` — ключ мерчанта.
- Заголовок `X-Request-Signature` — HMAC-SHA1 подпись тела запроса, Base64.

### 5.3. Реквизиты мерчанта
Каждая **точка (автомат)** должна иметь **отдельный `terminal_id`** (и, возможно,
отдельный `store_id`) — для разделения статистики и денег по точкам. Пример реального
формата реквизитов (из parking-проекта, для справки по структуре — **не использовать
эти значения напрямую**, для вендинга нужны свои):
```
merchant_id: 23174
store_id: 31494
terminal_id: 36051
api_key: <секрет, хранится только в .env>
```

### 5.4. Создание инвойса (аналог `create_invoice` из parking `jetqr.py`)
`POST /api/v1/merchant/invoice` — с `merchant_id`, `store_id`, `terminal_id`,
`mis_terminal_id`, `mis_payment_id` (уникальный, **должен содержать `machine_id` и
`slot_id`**, например `VND-{machine_id}-{slot_id}-{uuid}`), `mis_amount` (цена
конкретного товара, НЕ фиксированная, в отличие от parking).

### 5.5. Проверка статуса (аналог `check_invoice`)
`GET /api/v1/merchant/invoice?invoiceId={id}` — поллинг раз в 2 сек, как в parking
`poll_payment()`.

### 5.6. Отмена / возврат инвойса (НОВОЕ — этого нет в parking, нужно реализовать)
Инициируется мерчантом (нашим backend), JetQR пересылает в банк-эквайер.
- Полная отмена: `{ "invoice_id", "transaction_id", "cancellation_type": 0, "payment_method" }`
- Частичная отмена: `{ "invoice_id", "transaction_id", "cancellation_type": 1, "refund_amount", "original_amount", "new_amount" }`
- Статус отмены проверяется отдельным GET-запросом по `transaction_id`.
- **Важно:** точный merchant-facing URL для инициации отмены нужно уточнить у JetQR/Aliftech
  напрямую при подключении (в документации расписана реализация со стороны банка, а не
  конкретный эндпоинт для мерчанта).

## 6. Архитектура компонентов

```
[Клиент] --скан QR--> [Банк/кошелёк] --оплата--> [JetQR] <--поллинг-- [Центральный сервер]
                                                                              |
                                                          WebSocket (аутентифицированный,
                                                          по machine_id) на каждую точку
                                                                              |
                                                          [Контроллер точки: Pi/ARM]
                                                                    | RS232 (0x03/0x04)
                                                          [VMC автомата XY-Vending]
```

- **Центральный сервер** — один на все точки: FastAPI + PostgreSQL, JetQR-интеграция, админка.
- **Контроллер точки** ("Upper computer") — Pi/ARM-плата на каждом автомате:
  - Локально общается с VMC по RS232 (протокол из раздела 4, требует ответов за миллисекунды —
    **должен работать локально**, не через центральный сервер, т.к. по сети такую задержку
    не выдержать).
  - С центральным сервером общается на высоком уровне (WebSocket): "клиент выбрал слот N",
    "выдай слот N", "статус: успех/ошибка".

## 7. Идентификация автомата и привязка транзакции (важный вопрос, который разбирали)

1. **Регистрация**: при установке контроллера на точке ему один раз прошивается уникальная
   пара `machine_id` + секретный токен, хранится локально в `.env` на плате.
2. **Подключение**: контроллер подключается к серверу по `wss://server/ws/vending/{machine_id}`
   с токеном в заголовке. Сервер сверяет с таблицей `VendingMachine` в БД — если совпадает,
   принимает соединение и кладёт в словарь `{machine_id: websocket}` (аналог
   `exit_terminal_clients` из parking, но словарь, а не список — точек много).
3. **Привязка транзакции**: раз соединение уже аутентифицировано, всё что приходит по нему —
   однозначно от этого `machine_id`. При выборе товара создаётся `VendingSession` с полями
   `machine_id + slot_id + invoice_id` в одной строке.
4. **JetQR не знает про `machine_id`** — вся привязка "какой инвойс к какому автомату" живёт
   только в нашей БД. При подтверждении оплаты по `invoice_id` сервер смотрит сессию → достаёт
   `machine_id` → находит нужный WebSocket в словаре → шлёт команду выдачи только туда.

## 8. Схема данных (основные таблицы)

| Таблица | Ключевые поля | Назначение |
|---|---|---|
| `VendingMachine` | `machine_id` (PK), `name`, `location`, `secret_token`, `status` (online/offline), `owner_id` | Реестр точек сети |
| `ProductSlot` | `machine_id`, `slot_id`, `product_name`, `price`, `stock_qty` | Каталог товаров/цен по слотам (индивидуально на каждой точке) |
| `VendingSession` | `id`, `machine_id`, `slot_id`, `invoice_id`, `amount`, `status`, `created_at`, `closed_at` | Сессия покупки от выбора до выдачи/возврата |
| `Blacklist` | `phone_number`, `reason`, `added_at` | Блокировка недобросовестных клиентов (по номеру из ответа JetQR) |

Статусы `VendingSession.status`: `pending` → `paid` → `dispensing` → `dispensed` (успех)
либо `refund_pending` → `refunded` (сбой выдачи).

## 9. Обработка сбоя выдачи (refund-флоу)

1. Контроллер получает от VMC статус `0x04` с кодом ошибки (или не получает ответ в течение
   таймаута — тоже считаем ошибкой).
2. Сервер переводит `VendingSession.status = refund_pending`.
3. Сервер автоматически вызывает отмену инвойса в JetQR (раздел 5.6).
4. Клиенту на экране автомата — сообщение о технической ошибке и возврате средств.
5. Инцидент логируется с привязкой к `machine_id + slot_id` — для планирования техобслуживания
   (если конкретный слот часто фейлит — сигнал технику).

## 10. Функциональность (полный список из обсуждения — приоритизировать при разработке)

**Оплата и клиент:**
1. Единый динамический QR-код на каждую покупку (сумма = цена товара).
2. Автоматический возврат денег при сбое выдачи.
3. Электронный чек клиенту (по номеру телефона из ответа JetQR).
4. Частичная оплата / доплата другим способом.
5. Промокоды и скидки по времени/точке.
6. Программа лояльности — бонусы по номеру телефона.

**Управление точками (админка):**
7. Общая статистика по всем точкам города.
8. Статистика по каждой точке отдельно (топ товаров, выручка, часы пик).
9. Удалённое изменение цен без выезда на точку.
10. Карта города с онлайн/офлайн статусом точек в реальном времени.
11. Учёт остатков товара по слотам.
12. Выгрузка отчётов (Excel/CSV) по точкам/товарам/датам.

**Надёжность и алерты:**
13. Уведомление техника при неисправности автомата/потере связи.
14. Уведомление о нехватке товара на слоте.
15. Алерты в Telegram о критических событиях.
16. Журналирование частых сбоев по слоту.
17. Ручная выдача товара оператором с админки (форс-мажор).

**Масштабирование и безопасность:**
18. Мультиарендность — разные владельцы точек видят только свою статистику.
19. Чёрный список номеров телефонов при подозрении на мошенничество.
20. Многоязычный интерфейс (таджикский/русский/английский).

## 11. Технический стек (рекомендуемый, по аналогии с parking)

- Backend: **Python 3 + FastAPI**
- БД: **PostgreSQL + SQLAlchemy**
- Связь сервер↔точка: **WebSocket** (или MQTT при большом кол-ве точек с нестабильным интернетом)
- Контроллер точки: **Raspberry Pi / ARM-плата** + RS232-адаптер (USB-RS232 при необходимости)
- Секреты (JetQR API-ключи, токены точек) — только через `.env`, не в коде/репозитории

## 12. Термины

- **QR** — Quick Response, двумерный штрихкод для инициации платежа.
- **JetQR** — платёжный агрегатор единого QR-кода (Aliftech), приём оплаты с разных банков.
- **VMC** — Vending Machine Controller, главная плата управления автоматом.
- **Upper computer** — внешний контроллер (наш Pi/ARM), управляющий VMC по RS232.
- **Инвойс** — электронный счёт на оплату, формируется в JetQR под конкретную транзакцию.
- **Мостик / контроллер точки** — устройство на точке, связывающее RS232-шину VMC с центральным сервером.
